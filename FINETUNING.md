# 파인튜닝 계획 — slplab 음소 모델을 읽기평가에 맞게 적응

> 전제: 공개 모델 교체로는 정확도가 수렴함(slplab이 최선, A/B로 확인). 이 이상의
> 정확도 도약은 **파인튜닝**뿐. 여기서는 "어떻게"를 구체적으로 설계한다.

---

## 0. 가장 중요한 함정 — "오독을 고치게 만들면 안 된다"

읽기평가의 생명은 **오독을 오독 그대로 받아적는 것**(§2-3). 파인튜닝을 잘못하면
이 원칙이 깨진다.

- ❌ **하면 안 되는 것**: "아이가 [채상]이라 읽은 음성" + "정답 라벨 '책상'"으로 학습.
  → 모델이 틀린 소리를 정답으로 매핑하도록 배움 = self-correction. Whisper와 같은 문제.
- ✅ **해야 하는 것**: 라벨을 **실제로 발음한 음소**로 단다(책상을 [채상]으로 읽었으면
  라벨도 [채상]의 음소열). CTC라 외부 LM이 없으므로, 실제 음소로 학습하면
  **음향 충실도만 올라가고 보정 능력은 안 생긴다.**

이 한 가지가 데이터 라벨링 방식을 전부 결정한다.

---

## 1. 무엇을 고치려는 파인튜닝인가 (목표 2개)

| 목표 | 문제 | 데이터 성격 |
|---|---|---|
| **A. 화자 적응** | slplab은 성인 음성 학습. 아동은 음높이↑·조음 미숙·비유창 → 인식 저하 | 아동 '정상' 발음 음성(대량) |
| **B. 임상 정밀·보정** | 난독증 오독 패턴 충실 전사 + GOP 임계값 보정 | 아동이 검사문항 읽은 실제 음성 + SLP 음소 전사(소량·정밀) |

A는 대량 공개데이터로 가능, B는 임상 수집이 필요하지만 적은 양으로도 효과.

---

## 2. 데이터 전략 (2단계)

### 단계 1 — 아동 음향 적응 (대량, 라벨 자동)
- **출처**: AIHub 아동/유아 한국어 음성 데이터(읽기·발화). 대부분 **정상 발음**.
- **라벨 생성 (자동)**: 정상 발음이므로 *정답 텍스트 ≈ 실제 발음*. 따라서
  `g2pkk → phoneme_map`(우리 코드 그대로)으로 **목표 음소열을 자동 생성** → CTC 라벨.
- **효과**: 아동 음성 특성에 음향 적응. 임상 데이터 없이도 큰 향상 기대.
- **주의**: 정상 발음 데이터라 '난독증 오독 패턴'은 못 배움(그건 단계 2).

### 단계 2 — 임상 정밀 적응 + 보정 (소량, 라벨 정밀)
- **출처**: 아동이 우리 검사문항(무의미단어/음운규칙/일치 + 지문)을 읽은 **실제 녹음**.
- **라벨링 (사람)**: SLP가 **실제로 발음한 대로** 음소(또는 한글) 전사 → slplab 음소
  체계로 변환. 정상·오독 모두 포함.
- **이중 효과**:
  1. 모델이 비표준 발음을 충실히 전사하도록(보정 없이) 미세 적응.
  2. 이 SLP 전사가 곧 **GOP 임계값·채점 보정의 정답지**이자 **규준(percentile)의 원천**.
- **양**: 수백 발화만으로도 적응+보정 시작 가능(많을수록 좋음).

> 단계 2의 데이터는 정확도·캘리브레이션·규준을 **동시에** 해결한다(일석삼조).

---

## 3. 방법 (기술)

- **베이스**: `slplab/wav2vec2-xls-r-300m_phone-mfa_korean` (XLS-R 300M + CTC head).
- **손실**: CTC. 입력=16k 음성, 타깃=**실제 발음** 음소열(45-phone, 우리 체계 동일).
- **동결/학습**:
  - CNN feature encoder는 **동결**(표준 wav2vec2 파인튜닝).
  - Transformer + CTC head 학습. 데이터 적으면 **LoRA/adapter**로 파라미터 효율 학습
    (과적합·망각 방지).
- **vocab 유지**: 45 음소 그대로 → `phoneme_map`·정렬·GOP·채점 **코드 변경 0**.
  파인튜닝된 가중치만 갈아끼우면 드롭인 교체.
- **커리큘럼**: 단계1(대량) → 단계2(소량) 순차. 단계2는 작은 lr로.

---

## 4. self-correction 방지 검증 (필수 게이트)

- 학습/검증셋에 **일부러 틀린 읽기**(예: 책상→[채상]) 샘플을 넣고,
  파인튜닝 후에도 모델이 **[채상]을 그대로 출력**하는지 확인.
- 정답으로 보정되면 → 라벨링이 잘못된 것(정답 텍스트로 라벨함). 즉시 교정.

---

## 5. 평가 지표

| 지표 | 의미 |
|---|---|
| **PER**(음소 오류율) | 아동 held-out 음성에서 전사 정확도 |
| **SLP 일치도** | 자동 채점 vs SLP 채점 일치율 — **1순위**(브리프 §8). 특히 무의미단어 |
| **오독 검출 보존** | 일부러 틀린 셋에서 오독을 여전히 잡는지(보정 안 됨) |
| **GOP 분리도** | 정답/오독 음소의 GOP 분포가 임계값으로 잘 갈리는지 → 임계값 재설정 |

---

## 6. 단계적 릴리스

- **v0 (현재)**: slplab off-the-shelf + 규칙 채점 + GOP 주채점(잠정 임계값).
- **v1**: + 단계1(아동 음향 적응). 임상 데이터 불필요, 음향 정확도 향상.
- **v2**: + 단계2(임상 라벨). 오독 패턴 충실 + GOP 캘리브레이션 + 규준 구축 시작.

---

## 7. 리소스 개략

- **컴퓨팅**: 300M wav2vec2 파인튜닝 = GPU 1장(16~24GB)으로 단계당 수 시간.
  LoRA면 더 가벼움.
- **데이터**:
  - 단계1: 아동 음성 수십 시간(AIHub 확보 가능).
  - 단계2: 라벨 발화 수백~수천(임상 수집 + SLP 전사).
- **인력**: SLP 전사(단계2)가 병목이자 핵심 가치. 전사 가이드(음소 표기 규약) 필요.

---

## 8. 첫 스텝 (실행 순서 제안)

1. **AIHub 아동 음성 데이터 신청·확보** + 라이선스 확인.
2. 우리 `g2pkk→phoneme_map`으로 **자동 음소 라벨 생성 스크립트** 작성(코드 재사용).
3. **단계1 파인튜닝**(LoRA) → held-out PER로 향상 확인 + self-correction 게이트 통과.
4. 임상 파일럿: 아동 N명이 검사문항 읽기 → **SLP 음소 전사 규약** 정하고 소량 라벨.
5. **단계2 파인튜닝 + GOP 임계값 캘리브레이션** + 규준 데이터 축적 시작.

> 핵심: **단계1(공개 아동데이터, 라벨 자동)** 은 임상 데이터 없이 지금 바로 착수 가능.
> 가장 비용 대비 효과가 큰 첫 수.
